# 커밋(Commit)이란 무엇인가

본문 [§7 원칙 6 "대규모 작업은 체크리스트로 격파하라"](../../README.md#7-에이전트-통제술-part-2-작업-워크플로우)에서 "항목 하나 완료하면 체크 표시 후 커밋" 이라고 말했습니다. 이 "커밋" 이 무엇인지, 그리고 왜 작업 단위마다 커밋을 남기는 게 중요한지를 이 부록에서 짧게 풀어놓습니다.

---

## 한 줄 요약

**커밋은 "지금 이 상태를 저장한다"는 스냅샷입니다.**

작업하던 파일들의 변경 내용을 묶어서, 이름(메시지)을 붙여 기록하는 행위입니다. 한 번 커밋된 내용은 나중에 언제든 되돌아볼 수 있고, 이전 상태로 복구할 수도 있습니다.

---

## 비유로 이해하기

게임의 **세이브 포인트**와 같습니다.

- 게임에서 중요한 지점마다 저장해두면, 실수하거나 다른 시도를 해보고 싶을 때 그 지점으로 돌아올 수 있습니다.
- 커밋도 마찬가지입니다. 코드(또는 문서) 작업 중 "여기까지는 확실히 동작한다"는 시점에 커밋해두면, 이후에 무언가 잘못되더라도 이 지점으로 되돌릴 수 있습니다.

---

## Git과 커밋의 관계

커밋은 **Git**이라는 버전 관리 시스템의 핵심 개념입니다. Git은 파일의 변경 이력을 커밋 단위로 쌓아두는 도구입니다.

```
commit 1  "first draft"       <- 첫 초안
  |
commit 2  "add chapter 2"     <- 2장 추가
  |
commit 3  "fix typo"  <-- now <- 오탈자 수정, 현재 위치
```

이 커밋들이 쌓인 공간을 **저장소(repository, 줄여서 repo)** 라고 합니다.

---

## 로컬 커밋 vs 푸시(Push)

커밋은 기본적으로 **내 컴퓨터 안**에만 저장됩니다.

다른 사람과 공유하거나 GitHub 같은 원격 저장소에 올리려면 **푸시(push)** 라는 별도 동작이 필요합니다.

```
  My PC (local)                GitHub (remote)
+------------------+          +------------------+
| commit 1         |          | commit 1         |
| commit 2         |  push -->| commit 2         |
| commit 3 (new)   |          |                  |
+------------------+          +------------------+
```

로컬 커밋은 언제든 되돌리거나 수정할 수 있어서 부담이 없습니다. 푸시는 공유 공간에 올라가는 것이라 신중하게 합니다.

---

## Claude Code에서의 커밋

Claude Code는 작업이 완료된 시점에 자동으로 커밋을 만들 수 있습니다. 본문 §7 원칙 6의 "항목 하나 완료 → 커밋" 워크플로우는 다음을 의미합니다:

- 작업 단위가 명확하게 기록으로 남습니다.
- 무언가 잘못됐을 때 직전 커밋으로 즉시 되돌릴 수 있습니다.
- 세션이 바뀌어도 어디까지 했는지 커밋 이력으로 확인할 수 있습니다 — 다음 세션에서 `git log` 한 줄이면 맥락이 복원됩니다.

세이브 없이 긴 게임을 하다 죽으면 처음부터인 것처럼, 커밋 없이 긴 작업을 하다 실수하면 되돌릴 지점이 없습니다.

> 세션 인수인계는 커밋 이력 + [CONTINUE.md 패턴](../../README.md#7-에이전트-통제술-part-2-작업-워크플로우) 두 축으로 설계됩니다. 커밋이 "코드의 기록" 이라면 CONTINUE.md는 "의도·다음 단계의 기록" 입니다.
